## 3. Manual Verification of `www.lyzr.ai` Findings

Reverse-engineered the scanner's claims with read-only probes (curl GETs, no state
changes).

**Key discovery: scanner misidentified the tech stack.** lyzr.ai is WordPress +
Elementor + Kadence theme (not Next.js). This caused cascading false positives:

| Scanner Finding | Verified Status | Reason |
|-----------------|-----------------|--------|
| SSTI on `/` (CVSS 9.5) | **False positive** | `{{7*7}}` probe not reflected at all. Scanner match was on "49" appearing naturally in CSS/image dimensions. |
| SSTI on `/playbook/*` (CVSS 9.5) | **False positive** | Same cause. |
| SSTI on `/blueprints/*` ×2 (CVSS 9.5) | **False positive** | Same cause. |
| JWK trust on `/api/event` (CVSS 9.5) | **False positive** | `/api/event` is an Apache URL rewrite → 301 → `/event-info/` marketing page. Not an API at all. Scanner treated HTML response after redirect as "server accepted JWT". |
| Next.js `/_next/data/` exposed (CVSS 7.5) | **False positive** | Site is WordPress, not Next.js. |
| DOM XSS patterns (CVSS 6.1) | **Partially true** | jQuery 3.7.1 is patched. But real postMessage gadgets were present. |
| Weak CSP | **Confirmed** | CSP is literally just `upgrade-insecure-requests`. |
| 13 scripts without SRI | **Confirmed** | Many CDN trackers without integrity checks. |

**Real issues the scanner missed** (found during verification, in inline HTML scripts):

1. **HubSpot tracking cookie leak via postMessage** — listener responds to
   `REQUEST_HUBSPOT_TOKEN` messages with no `event.origin` check, sends the cookie
   back via `event.source.postMessage(..., '*')`. Exploitable via popup window
   (X-Frame-Options: SAMEORIGIN blocks iframes but not `window.open`).
2. **Open redirect** — `OPEN_BOOKING_LINK` handler does
   `window.open(data.url, '_blank')` with no URL validation.
3. **GTM event injection** — attacker-controlled `data.event` and `data.email`
   pushed into `window.dataLayer`.
4. **`new Function(fb_pxl_code)` sink** — from Meta Pixel WordPress plugin.
   Backend-sourced, not directly exploitable today, but dangerous pattern.

**Honest re-score for lyzr.ai: probably a C, not an F.** No confirmed criticals
(scanner was wrong about all 5). Real medium-severity postMessage issues the
scanner didn't find. Expected WordPress hardening gaps.

## 4. Scanner Fixes Shipped

Reliability fixes in response to the lyzr.ai false positives:

| Fix | File | Change |
|-----|------|--------|
| SSTI probe | `modules/injection.ts` | Replaced `{{7*7}}`→49 / `{{8*8}}`→64 probes with `{{19123*19321}}`→369475483 and `{{27351*28121}}`→769137471. Requires baseline to lack both markers and both probes to succeed. 9-digit products can't appear in image dimensions / CSS percentages / dates by chance. |
| JWK trust check | `modules/jwt-check.ts` | Reject when `res.url` pathname differs from requested (redirect to different resource). Reject HTML responses entirely — a JWT-accepting API returns JSON/text, never a full HTML page. |
| Tech detection | `modules/recon.ts` | Next.js markers narrowed to `__NEXT_DATA__`, `_next/static/`, `next/dist/`. Header detection requires `x-nextjs-*` header or literal `next.js` token (not loose `/next/i`). Added WordPress + Elementor signatures. When WordPress is detected, strips Next.js / Remix / Nuxt / SvelteKit / Astro — mutually exclusive stacks. |
| New module | `modules/postmessage.ts` | Parses `addEventListener('message', ...)` handler bodies from inline scripts + bundled JS. Flags handlers missing `event.origin` validation that contain dangerous sinks (`window.open(data.url)`, `innerHTML = data.*`, `eval`/`new Function` with message data, cookie reads followed by postMessage reply). Confidence 70. |

Module count is now **55** (was 54). Updated `app/layout.tsx` metadata.

Verified: type-check clean, `npm test` 42/42, `npm run build` clean.
